| 1 | HIGH | Security / False-positive | tasks.md T-02, design.md D1/D2 | **In-process anchor is never seeded for resumed processes — per-node on-disk check breaks on every resumed first-node.** T-06 check (b) compares `SHA256(full on-disk events.jsonl)` against the in-process anchor. For a fresh run (single process), the anchor accumulates from an empty base and at any point matches the full file — correct. For a **resumed process** (new Bun process, existing journal on disk), `JobJournal.persist` first reads `existingCounters` from state.json, then appends only the **delta records** since those counters. The in-process registry, which T-02 says "accumulates in lockstep with `appendEventRecord`", starts empty and accumulates only delta bytes. After `begin()`, the in-process anchor holds only the begin-era delta (a few lines), while `events.jsonl` on disk holds the full history from all prior sessions. `SHA256(full file) ≠ SHA256(delta)` → **per-node check (b) always fails for the first resumed node** (false positive, violates T6 for resume paths). Alternatively, if the implementation skips check (b) when in-process anchor has only delta bytes, **T1/T2 tampers during the first resumed node go undetected** (security gap). D2's "defers to durable anchor" does not resolve this: after `begin()` appends new records, `SHA256(on-disk)` > `SHA256(durable anchor)`, so durable-anchor comparison also fails. | Add explicit anchor seeding to the spec and tasks. Recommended: in `JobJournal.persist`, when `existingCounters` is non-null (i.e., the journal file already exists), **read the full on-disk events.jsonl and state.json once before writing any new bytes**, seed the registry with those bytes, then append the delta. This is a one-time read **before** the write (D1's "never re-read after a write" is not violated). T-02 should state this seed logic explicitly. Alternatively, the spec may choose to: (a) skip check (b) on fresh-process resumed nodes and document the resulting first-node tamper window, or (b) use the durable anchor as the baseline for check (b) on fresh-process resumed nodes (requires re-fetching origin before each per-node check). Whichever is chosen, make it explicit in T-02, T-06, and the Risks section. |
| 2 | MEDIUM | Spec ambiguity | tasks.md T-04, design.md D5 | **Pipeline journal commit placement relative to `commitSuccess` leaves success-era bytes durably unanchored per-node.** T-04 says "wire the pipeline-managed journal commit into the sequential finalize path **after per-node verification (T-06) passes**." T-06 says verification runs "**between `produce` and `apply`**". This places the pipeline journal push before `orchestrator.apply()` → `commitSuccess()`. But `commitSuccess` appends two more journal records (verdict history entry + success persist). Those success-era bytes are not included in the per-node push; they are only pushed at the next node's journal commit or at checkpoint (`commitFinalState`). The Risks section acknowledges the crash window (spurious halt on resume, "accepted P0 posture"), but the tasks do not document this ordering decision explicitly. An implementer reading T-04 in isolation may attempt to include success-era bytes in the per-node journal push — which would require the orchestrator (B-13 single-writer) to perform a git side effect, violating the B-13 invariant. | Add a note to T-04 (or a separate paragraph in design.md D3/D5) explicitly stating: "The per-node journal push captures journal bytes through begin's persist only; success-era bytes (verdict history + success persist from commitSuccess) are pushed at the next node's journal commit or at checkpoint. This is intentional and consistent with the accepted P0 posture documented in Risks." Also clarify in T-09 / T6 that the false-positive-fixation scenario tests a clean run with no crash and an intentional `awaiting-resume` pause — not crash recovery. |
| 3 | MEDIUM | Spec completeness | design.md Open Questions | **Two Open Questions in design.md are unresolved and have implementation-correctness implications.** (a) "Durable ref object form and cadence" — whether the object is a blob/tag/commit and the exact ref namespace is left to the implementer. Different choices have different TOCTOU and race properties; e.g., a lightweight tag vs a blob ref behaves differently under concurrent pushes. (b) "Round-path journal commit sweep" — whether the coordinator round needs its own post-round journal commit or can rely on the next sequential step's push. If the pipeline ends inside a round (e.g. last step is a parallel code-review round) and there is no subsequent sequential node to push the round's journal, origin may be missing those journal bytes at checkpoint push time, making the durable anchor incomplete. | Resolve both Open Questions before implementation. (a) Specify the ref as a git blob ref (simplest, no history): `git update-ref refs/specrunner/evidence/<branch> <blob-oid>` where the blob encodes the digest string. Document this in design.md D4 and T-03. (b) State explicitly: "if the final step before a terminal transition is a round, the coordinator emits a journal commit after `commitRound` completes, before `commitFinalState`." This prevents an origin durability gap. |
| 4 | LOW | Spec clarity | tasks.md T-09/T6, design.md Risks | **T6 acceptance criterion mixes two different "clean resume" meanings.** T6 requires "a clean resume … assert no authenticity halt fires." The Risks section simultaneously accepts that a crash between a per-node journal push and the next `commitSuccess` causes a spurious halt on resume ("accepted P0 posture"). These two statements are only consistent if "clean resume" means "resume from an intentional `awaiting-resume` pause" (not from a mid-node crash). The distinction is not explicit. A test author implementing T6 who writes a crash-then-resume scenario may falsely expect it to pass without a halt. | Qualify T6 in tasks.md as: "clean resume = resume from an intentional `awaiting-resume` checkpoint pause (not a crash-recovery resume). Crash-recovery spurious halts are accepted P0 posture (see design.md Risks)." |
| 5 | LOW | Spec clarity | tasks.md T-06, design.md D2 | **"Absent anchor-with-present-on-disk that cannot be evaluated is a violation" conflicts with D2's fail-open for absent anchor.** T-06 fail-closed rule says "absent anchor-with-present-on-disk that cannot be evaluated is a violation." D2 says "absence preserves current behavior (no baseline → no check)". These are contradictory unless "cannot be evaluated" is scoped narrowly (e.g., means BOTH in-process AND durable are absent). The current phrasing allows two opposing implementations depending on which document the implementer prioritises. | Unify in T-06: "If BOTH the in-process anchor AND the durable anchor are absent for an existing on-disk journal, treat as tamper (fail-closed). If only the in-process anchor is absent but the durable anchor is present, use the durable anchor as the check baseline (see Finding #1). If both are absent and the journal is also empty/absent (brand-new job, first write), skip check — no baseline exists yet." |
