## Requirements

### Requirement: pr-create-runner spawns gh CLI to create or detect PRs

A CLI runner module SHALL be implemented at `src/core/pr-create/runner.ts` exposing the function:

```ts
export async function runPrCreate(input: {
  branch: string;
  baseBranch: string;
  title: string;
  body: string;
  cwd: string;
}): Promise<PrCreateRunnerResult>;

type PrCreateRunnerResult =
  | { status: "created"; url: string; number: number }
  | { status: "existing-open"; url: string; number: number }
  | { status: "error"; reason: "merged" | "closed" | "gh-failure"; message: string };
```

The runner SHALL execute the following sequence:

1. Spawn `gh pr list --head <branch> --base <baseBranch> --state all --json url,number,state` (with `cwd: input.cwd`) to detect existing PRs for the branch. This command always exits with status 0 and returns a JSON array (empty array when no PR exists).
2. If the JSON array has length 0: no PR exists — proceed to step 5
3. If the JSON array has length >= 1: inspect the first element's `state` field:
   - `"OPEN"`: return `{ status: "existing-open", url, number }` (idempotent path)
   - `"MERGED"`: return `{ status: "error", reason: "merged", message: <descriptive> }`
   - `"CLOSED"`: return `{ status: "error", reason: "closed", message: <descriptive> }`
4. If the `gh pr list` command exits with non-zero status: return `{ status: "error", reason: "gh-failure", message: <stderr> }`
5. Spawn `gh pr create --title <title> --body-file <tempfile> --base <baseBranch> --head <branch>` (with `cwd: input.cwd`), where `<tempfile>` is a temporary file written with `body` content via `fs.writeFile()`. The `--body` flag SHALL NOT be used (argv size risk for large bodies). The tempfile SHALL be deleted after the command completes (success or failure).
6. On exit status 0: parse the stdout to extract `url` and `number` and return `{ status: "created", url, number }`
7. On any other gh CLI failure: return `{ status: "error", reason: "gh-failure", message: <stderr> }`

The runner SHALL NOT use stderr message matching to detect "no PR found". PR absence MUST be determined solely by the JSON array length from `gh pr list`. The runner SHALL NOT retry on failure. Any retry strategy is the responsibility of the caller (currently `PrCreateStep`, which does not retry).

The runner MUST NOT depend on the `GitHubClient` port introduced in PR #34. The `gh` CLI is a separate I/O channel from the GitHub REST API used by other parts of the system.

#### Scenario: Existing OPEN PR is detected and returned without re-creation

- **GIVEN** branch `feat/foo` has an existing OPEN PR `https://github.com/owner/repo/pull/12` on GitHub
- **WHEN** `runPrCreate({ branch: "feat/foo", baseBranch: "main", ... })` is invoked
- **THEN** `gh pr list --head feat/foo --base main --state all --json url,number,state` is spawned
- **AND** the JSON output is `[{ "url": "https://github.com/owner/repo/pull/12", "number": 12, "state": "OPEN" }]`
- **AND** the result is `{ status: "existing-open", url: "https://github.com/owner/repo/pull/12", number: 12 }`
- **AND** `gh pr create` is NOT spawned

#### Scenario: New PR is created when none exists

- **GIVEN** branch `feat/bar` has no existing PR
- **WHEN** `runPrCreate({ branch: "feat/bar", baseBranch: "main", title: "Add bar", body: "...", cwd: "/path/to/repo" })` is invoked
- **THEN** `gh pr list --head feat/bar --base main --state all --json url,number,state` returns JSON output `[]`
- **AND** `gh pr create --title "Add bar" --body-file <tempfile> --base main --head feat/bar` is spawned in `cwd`
- **AND** the result is `{ status: "created", url: <new url>, number: <new number> }`
- **AND** the tempfile is deleted after the command completes

#### Scenario: MERGED existing PR returns error

- **GIVEN** branch `feat/baz` has an existing MERGED PR
- **WHEN** `runPrCreate({ branch: "feat/baz", baseBranch: "main", ... })` is invoked
- **THEN** `gh pr list --head feat/baz --base main --state all --json url,number,state` returns `[{ ..., "state": "MERGED" }]`
- **AND** the result is `{ status: "error", reason: "merged", message: <descriptive> }`
- **AND** `gh pr create` is NOT spawned

#### Scenario: gh CLI failure returns error with stderr

- **GIVEN** the `gh` CLI is not authenticated (`gh auth status` would fail)
- **WHEN** `runPrCreate({ ... })` is invoked
- **THEN** the result is `{ status: "error", reason: "gh-failure", message: <stderr from gh> }`
- **AND** the message includes a hint pointing to `specrunner login` or `gh auth login` for re-authentication

### Requirement: pr-create-result.md is generated by PrCreateStep with structured Status

`PrCreateStep.run` SHALL write a markdown file at `resultFilePath(state)` containing a top-level Status marker (`Status: success` or `Status: failed`), a PR section listing url / number / branch / createdAt, and a Detail section with a diagnostic message. `Status: success` SHALL be written when the runner returns `status: "created"` or `status: "existing-open"`. `Status: failed` SHALL be written when the runner returns `status: "error"`.

#### Scenario: pr-create-result.md is created on success

- **GIVEN** `PrCreateStep.run` invokes the runner and receives `{ status: "created", url: "<u>", number: 7 }`
- **WHEN** `run` finishes
- **THEN** the file at `resultFilePath(state)` contains the line `## Status: success`
- **AND** the file contains the URL and number under the `## PR` section

#### Scenario: pr-create-result.md is created on failure

- **GIVEN** `PrCreateStep.run` invokes the runner and receives `{ status: "error", reason: "gh-failure", message: "auth expired" }`
- **WHEN** `run` finishes
- **THEN** the file at `resultFilePath(state)` contains the line `## Status: failed`
- **AND** the `## Detail` section includes `gh-failure` and `auth expired`

### Requirement: PR body template is generated from request.md and pipeline state

A pure helper SHALL be defined at `src/core/pr-create/body-template.ts` exposing:

```ts
export function renderPrBody(input: {
  parsedRequest: ParsedRequest;       // request.md sections (背景, 目的)
  jobState: JobState;                 // for workflow phase summary
}): string;

export function renderPrTitle(parsedRequest: ParsedRequest): string;
```

`renderPrTitle` SHALL return the level-1 heading (`#`) of the request.md verbatim.

`renderPrBody` SHALL produce a markdown string containing:

1. A `## Summary` section composed of the request.md `## 背景` and `## 目的` sections concatenated (verbatim text, no LLM summarization)
2. A `## Workflow` section containing a markdown table with one row per workflow phase. Each row SHALL include: phase name (`spec-review` / `verification` / `code-review`), final iteration verdict, iteration count, and the result-file path of the latest iteration. Phases that did not execute SHALL be omitted
3. A `## Test plan` section that lists, for each phase that wrote a result file, a checkbox referencing the result-file path
4. A trailing line `🤖 Generated with SpecRunner` as the signature

The helper SHALL be pure (no I/O, no side effects). All inputs SHALL be passed in via the input object.

#### Scenario: renderPrTitle returns the request.md heading

- **GIVEN** a `parsedRequest` whose `title` is `"pr-create step 追加（self-host pipeline 完成形）"`
- **WHEN** `renderPrTitle(parsedRequest)` is invoked
- **THEN** it returns `"pr-create step 追加（self-host pipeline 完成形）"`

#### Scenario: renderPrBody includes Summary, Workflow, Test plan, and signature

- **GIVEN** a `parsedRequest` containing `## 背景` and `## 目的` sections, and a `jobState` with completed `spec-review`, `verification`, and `code-review` step runs
- **WHEN** `renderPrBody({ parsedRequest, jobState })` is invoked
- **THEN** the returned string contains the `## Summary` heading followed by the verbatim text of `## 背景` and `## 目的`
- **AND** the string contains a `## Workflow` table with rows for `spec-review`, `verification`, and `code-review`
- **AND** the string contains a `## Test plan` section with at least one checkbox
- **AND** the string ends with the line `🤖 Generated with SpecRunner`

#### Scenario: renderPrBody omits phases that did not run

- **GIVEN** a `jobState` where `code-review` produced 0 step runs
- **WHEN** `renderPrBody({ ... })` is invoked
- **THEN** the `## Workflow` table does NOT contain a `code-review` row

#### Scenario: renderPrBody includes Fixes line when parsedRequest.issue is set

`renderPrBody` は `parsedRequest.issue` が存在する場合、Summary section の直後に `Fixes ${issue}` 行を挿入する。issue が `undefined` のとき挿入しない。

- **GIVEN** `parsedRequest.issue === "#279"`
- **WHEN** `renderPrBody({ parsedRequest, jobState, slug })` is invoked
- **THEN** the body contains `Fixes #279`

#### Scenario: renderPrBody excludes Fixes line when issue is absent

- **GIVEN** `parsedRequest.issue === undefined`
- **WHEN** `renderPrBody({ parsedRequest, jobState, slug })` is invoked
- **THEN** the body does NOT contain any `Fixes #` line

### Requirement: PR creation uses REST API instead of gh CLI

The `runPrCreate()` function SHALL use the injected `GitHubClient` port for PR operations instead of spawning `gh` CLI subprocesses.

#### Scenario: PR listing via REST API
- **WHEN** `runPrCreate()` checks for existing PRs
- **THEN** it calls `GitHubClient.listPullRequests()` instead of `gh pr list`

#### Scenario: PR creation via REST API
- **WHEN** `runPrCreate()` creates a new PR
- **THEN** it calls `GitHubClient.createPullRequest()` instead of `gh pr create`
- **AND** no temporary file is created for the PR body (body is passed directly in the API request)

### Requirement: PrCreateInput accepts GitHubClient

The `PrCreateInput` interface SHALL accept `githubClient`, `owner`, and `repo` fields.

#### Scenario: Input contract
- **WHEN** `runPrCreate()` is called
- **THEN** the `githubClient`, `owner`, and `repo` fields are required
- **AND** the `githubToken` field is no longer accepted
