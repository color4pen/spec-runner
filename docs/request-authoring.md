# request の書き方

request.md は pipeline への唯一の入力であり、その解像度が pipeline の精度を決める。agent は推測の余地が少ない問題ほど正確に解く。このガイドは request の品質を作る具体的な技法をまとめる。

雛形は `specrunner request template`、構文検査は `specrunner request validate <file>` で得られる。

## 設計要素引用 — 設計レイヤとの紐付け（任意）

設計レイヤ CLI（aozu）を導入しているプロジェクトでは、`## 設計要素引用` セクションに、この request が実装する設計要素の `[[id]]` を列挙できる。

```markdown
## 設計要素引用

<!-- この request が実装する設計要素の [[id]] を列挙してください -->

[[mod-intake]], [[ent-order]]
```

aozu は request 本文全体から `[[id]]` 形式の引用を抽出して実在解決・状態検証を行う。セクション名や位置は緩く、本文のどこに書いても認識される。

このセクションは**任意**である。設計レイヤを使用していないプロジェクトは省略してよい。設計レイヤが有効なプロジェクトで引用が必要な場合、`request validate` と `run` の preflight で `aozu check` が自動実行される（`.specrunner/config.json` の `designLayer.enabled` が `true` のとき）。

## 現状コードの前提 — 書く直前に検証する

「今のコードはこうなっている」という断定は、必ず file:line を伴って書く。そして**書く直前に grep / Read で実コードと突き合わせる**。過去の調査メモや記憶から書いた前提は、その後の merge で腐っていることがある。腐った前提は design / request-review の照合で escalation になり、往復が一回無駄になる。

- 良い: 「fixer → review の逆引きは `.find()` で最初の対を返す実装であり（`src/core/pipeline/pipeline.ts:357-358`）、多対一を想定していない」
- 悪い: 「逆引きは多対一に対応していないはず」

前提の節は「未検証の主張」として design が実コードと照合する契約になっている。検証済みの前提だけを書けば、照合は確認作業で済む。

## 要件 — 最重量部を行間に隠さない

実装の中で一番重い変更（既存機構の一般化、暗黙の前提の変更、複数箇所への波及）は、要件として**名指しで**書く。受け入れ基準の行間から implementer が読み取ることを期待しない。

重い変更を名指しするには、起票前にその変更がどこに波及するかを一度歩く必要がある。この一手間が「実装してみたら設計が違った」という最も高い手戻りを防ぐ。

## 粒度 — 1 request は 1 つの収束ループ

1 request の規模は「1 つのレビュー収束ループで直しきれる範囲」を上限の目安とする。それを超える疑いがあれば分割のサインと考える。

分割の定石は**土台 → 上物**:

- **土台**: 機構の導入。「既定では挙動完全一致」を受け入れ基準に持つ（既存テストが無変更で green）
- **上物**: その機構の利用者。土台の merge 後に起票し、複数の上物は並列開発できる

分割の判定基準:

- 独立して設計・テストできる → 切る
- 収束の意味論が異なる（開放的レビュー vs 台帳照合、など）→ 必ず切る
- 片方が他方の受け入れ基準を参照する → 切らない（同一 request）

## 受け入れ基準 — 機械検証できる文にする

基準は判定者（conformance）が機械的に真偽を確かめられる形で書く。

- 良い: 「〜であることをテストで固定する」「既存テストが無変更で green」「`typecheck && test` が green」
- 悪い: 「〜が適切に動作する」「〜が改善される」

「既存テストが無変更で green」は土台系 request の最重要基準で、挙動不変の主張を機械検証に変換する。

## 設計判断 — 却下した代替案と理由を書く

`architect 評価済みの設計判断` の節には、採用した判断だけでなく**却下した代替案とその理由**を書く。implementer が却下済みの案を再発明することと、reviewer が決着済みの論点を再提起することの両方を防ぐ。

- 例: 「reviewer ごとの専用 fixer は収束ループの組み合わせ爆発を招くため却下し、findings の出所識別で代替する」

## スコープ外 — 隣接する誘惑を名指しで切る

実装中に「ついでにやりたくなる」隣接変更を列挙する。スコープ外の列挙は implementer の逸脱を防ぐと同時に、レビューの判定基準（この変更は逸脱か否か）にもなる。

## type の選択

設計の追加・変更を含むなら bug-fix を選ばない（spec-change / new-feature）。type は step のモデル選択（byRequestType）と reviewer の判定姿勢に影響する。adr は雛形コメントの判断基準に従う。

## 運用 — 議論の到達点をその場で書き落とす

設計議論の結論は、会話の中に置いたままにせず、その場で draft（`specrunner/drafts/`）または issue に書き落とす。drafts/ のファイルの存在は「未決のスレッド」、不在は「決着」を意味する。書き落とされた到達点は次のセッション・次の書き手の原本になる。

issue を実行可能にする場合、issue 本文がそのまま request.md になる（テンプレート構造をそのまま使う）。承認ラベルを付けるまで実行されないため、起票と実行判断は分離できる。
